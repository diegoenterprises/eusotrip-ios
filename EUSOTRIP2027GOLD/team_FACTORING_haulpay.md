# FACTORING User Role — EusoTrip Mobile App Doctrine 2027

**Wave 1 dossier.** This doctrine section is canonical. It governs the twenty-fourth named persona in EusoTrip's role lattice: **FACTORING**. It is distinct from DRIVER, DISPATCH, BROKER, SHIPPER, CATALYST, ESCORT, TERMINAL_MANAGER, COMPLIANCE_OFFICER, and SAFETY_MANAGER in the Trucking 12-role family, and it is the operational counterpart to the HaulPay integration that threads through the wallet, settlement, and load lifecycle.

This dossier is grounded in the HaulPay API documentation at docs.haulpay.io (Production base URL `https://api.haulpay.io/v1/external_api/`, Staging `api-staging.haulpay.io`, Bearer-token authentication) and in the backend router at `frontend/server/routers/factoring.ts` plus the role catalogue in `frontend/server/_core/trpc.ts`.

---

## 1. The FACTORING Role in EusoTrip's 24-Role System

The `ROLES` constant exported from `frontend/server/_core/trpc.ts` enumerates twenty-four personas across three transport-mode families:

- **Trucking (12):** SUPER_ADMIN, ADMIN, SHIPPER, CATALYST, BROKER, DRIVER, DISPATCH, ESCORT, TERMINAL_MANAGER, **FACTORING**, COMPLIANCE_OFFICER, SAFETY_MANAGER.
- **Rail (6):** RAIL_SHIPPER, RAIL_CATALYST, RAIL_DISPATCHER, RAIL_ENGINEER, RAIL_CONDUCTOR, RAIL_BROKER.
- **Vessel (6):** VESSEL_SHIPPER, VESSEL_OPERATOR, PORT_MASTER, SHIP_CAPTAIN, VESSEL_BROKER, CUSTOMS_BROKER.

FACTORING is an **operator-class** role — not a cargo-carrying persona. The role constant is exported as `ROLES.FACTORING`, and the tRPC layer already ships pre-built procedures for it: `factoringProcedure`, `auditedFactoringProcedure`, and `isolatedFactoringProcedure` (the last is the gold-standard variant that adds approval gating, isolation context, and auto-audit logging).

### Who Occupies This Role

Three populations authenticate as FACTORING:

1. **HaulPay account managers** working on behalf of the integrated factor. They act through API keys provisioned per tenant, and their user records carry `metadata.externalProvider = "haulpay"` so audit logs distinguish their mutations from in-house staff.
2. **EusoTrip's in-house factoring desk.** Human operators who sit inside EusoTrip's Ops organization, hired to review invoices HaulPay flagged as exceptions, service large accounts, and run the aging report each morning.
3. **Third-party factoring partners** for carriers who contract outside HaulPay. These partners use EusoTrip as a system-of-record only; they do not call the HaulPay API but they still submit manual advances that reconcile through `factoring.submitInvoice`.

### Permissions

- **View:** every invoice in the tenant scope that is awaiting an advance, regardless of which carrier submitted it.
- **Approve / Deny:** state transitions from `submitted` → `under_review` → `approved` → `funded`, and the rejection path into `disputed` or `chargedback`.
- **Trigger payouts:** issue the HaulPay quickpay request that credits the driver wallet, or run the weekly batch ACH.
- **Cannot create loads, bid on loads, or dispatch.** FACTORING is a money-movement role; it is firewalled from operational routers (`loads`, `bids`, `dispatch`). Attempting a `loads.create` from a FACTORING-roled session returns HTTP 403 with the RBAC_VIOLATION audit event.

### Admin Sub-Types

Sub-tiers are expressed via a `factoringTier` scope attribute in `users.metadata` (not a separate role, to avoid ballooning the 24-role lattice):

- **super-admin-factoring:** unlimited approval authority, access to risk-policy editor, reserve-ratio changes, and chargeback write-offs. Always a SUPER_ADMIN with `factoringTier = "super"`.
- **tier-1 account manager:** approve advances less than $10,000 per invoice, up to $50,000/day aggregate.
- **tier-2:** any amount within the tenant's active credit limit. Required for invoices above $10K.

The tier check runs *inside* `factoring.approve` after the RBAC gate; a tier-1 user attempting a $12,000 approval receives `TRPCError` code `FORBIDDEN` with message "Tier-2 approval required for this invoice amount."

## 2. HaulPay API Contract — Integration Points

### Authentication

Per docs.haulpay.io: the API is REST, JSON-in / JSON-out, and uses **Bearer token authentication**. The header is:

```
Authorization: Bearer 31ada4fdadec460c809a9e56ceb75269
```

The key is generated by the broker dashboard at app.haulpay.io and shared with the integrated TMS (EusoTrip, in our case). Each tenant that factors through HaulPay must register its own key — do not share a single EusoTrip-global key. Keys are stored encrypted at rest (via the `sensitiveData.encrypt` helper) and never logged.

No HMAC signing is documented for outbound API calls — Bearer is the sole auth. HMAC-SHA256 is reserved for **inbound webhooks** (see §10).

### Environments

- Production: `https://api.haulpay.io/v1/external_api/`, dashboard `https://app.haulpay.io`.
- Staging: `https://api-staging.haulpay.io/v1/external_api/`, dashboard `https://app.staging.haulpay.io`.

EusoTrip staging always points at HaulPay staging; a misrouted call (prod EusoTrip pointing at staging HaulPay) is a P1 deploy incident.

### Carrier / Debtor Enrollment

HaulPay's model is relationship-oriented. Their onboarding guidance reads:

1. Maintain our own list of debtors and carriers that the shared client actively uses.
2. Store the HaulPay **external ID** for each party.
3. Query `carrier_relationships_list` and `debtor_relationships_list` once per day per shared client to keep status fresh.
4. When the user wants to add a new party, search `carrier_list` or `debtor_list` first, surface results to the user, then POST to `carrier_relationships` or `debtor_relationships` with the external ID.
5. If the party does not exist in HaulPay, deep-link the user into HaulPay's "create carrier" / "create debtor" screens; creation through the external API is on their planned-expansions list but not yet live.

KYC and bank linkage are **owned by HaulPay**, not EusoTrip. Plaid bank connection, EIN verification, and identity documents flow through HaulPay's onboarding. EusoTrip stores only the HaulPay `carrierStatus` flag (active, pending, declined) so gating the "Get paid today" button is a single boolean check.

### Invoice Upload

The invoice workflow is: driver completes a load → Bill of Lading (BOL) and Proof of Delivery (POD) are captured and stored in our document service → backend assembles a factoring-invoice payload and POSTs it to HaulPay along with document URLs. HaulPay verifies document integrity, the debtor's credit status, and the lane, and returns an advance-eligibility object with `advanceRate`, `factoringFeePercent`, and `reserveAmount`.

Our wrapper (`factoring.submitInvoice`) mirrors this contract: `loadId`, `customerId`, `invoiceAmount`, `documents: [{ type: "bol" | "pod" | "invoice" | "rate_con", documentId }]`, and a `quickPay` boolean. When `quickPay === true`, the fee rate jumps from the standard `FACTORING_STANDARD` rate (default 2.5%) to `FACTORING_QUICKPAY` (default 3.0%).

### Quickpay Request

Driver experience: in the Wallet screen, the "Get paid today" CTA on the Factoring Offer card (shipped in iOS builds 54 and 55) calls `EusoTripAPI.shared.factoring.accept(loadId:offerId:)`. The backend then calls the HaulPay advance endpoint with the offer context; HaulPay moves funds to the carrier's bank of record (Plaid-linked at onboarding) and replies with `fundingEta` and a `transactionId`. The driver sees the advance reflected in the wallet within four hours for standard quickpay, same-day ACH-cutoff dependent.

### Status Webhooks

HaulPay calls back to EusoTrip on the following events (per the webhook sidebar in the docs):

- `invoice.approved`
- `invoice.funded`
- `invoice.collected` (debtor paid)
- `invoice.short_paid`
- `invoice.disputed`
- `invoice.chargedback`
- `carrier_relationship.status_changed`

Every callback hits `/api/webhooks/haulpay` and is routed into the factoring router's status-transition code. The router then updates `factoringInvoices.status`, writes a ledger entry via the payments router, and fires a push notification to the driver if the event affects them.

### Rate Card

HaulPay's published bps range is **250–500 bps (2.5%–5.0%)** for factoring fees on standard advances, with quickpay adding 50 bps typical. EusoTrip's default is stored in `platformFeeConfigs` under the `FACTORING_STANDARD` fee code with `baseRate = 0.025`, and is overridable per carrier via `userFeeOverrides`. The environment variable `FACTORING_FEE_BPS` serves as the bootstrap for a fresh tenant install (250 = 2.5%).

### Aging

HaulPay exposes an invoice-listing endpoint with filters for `submittedAt` ranges. The factoring desk dashboard pulls this daily and buckets into 30 / 60 / 90 / 90+ days. Anything past 60 days is flagged for collection; past 90 is eligible for chargeback to the carrier's reserve.

## 3. Backend Wiring

### `frontend/server/routers/factoring.ts`

This is the canonical tRPC router that wraps HaulPay. Key verified procedures (lines 279–500+):

- `factoring.create` — admin-procedure that creates a local `factoringInvoices` row, computes `feeAmount`, `advanceAmount`, and `reserveAmount` from the net-after-platform-fee settlement amount, and sets status to `submitted`.
- `factoring.update` — admin-procedure that transitions status. When status moves to `funded`, it writes a `platformRevenue` row under `transactionType = 'factoring_advance'` so the fee flows into the revenue ledger.
- `factoring.getOverview` — driver-facing query that returns account status, credit limit, used/available credit, reserve balance, factoring rate, and a recent-activity feed.
- `factoring.getInvoices` — paginated list of factored invoices for the current user, filtered by status.
- `factoring.submitInvoice` — the main submission procedure; accepts document references and a `quickPay` flag, writes the local row, returns an `estimatedFundingTime` ("4 hours" quickpay / "24 hours" standard).
- Internal helper `computeCreditScore` — a 300–850 platform-internal score that pulls FMCSA safety data (BASICs, safety rating), platform payment history (delivered loads, factoring invoice payment speed, short-pays, chargebacks), and company age. This score is what HaulPay does not directly provide and is the value-add layer EusoTrip contributes on top of HaulPay's own debtor credit.
- Rate/credit-limit lookup helpers `getFactoringConfig` and `getCreditLimit` — both query `platformFeeConfigs` with optional `userFeeOverrides` for per-carrier discounts.

Invoice status enum (line 274):
`submitted | under_review | approved | funded | collection | collected | short_paid | disputed | chargedback | closed`.

### `frontend/server/routers/payments.ts`

Every successful HaulPay advance writes two ledger entries: (a) a `wallet.credit` to the driver/carrier for `advanceAmount`, and (b) a `fee.debit` equal to `factoringFeeAmount` tagged with the invoice number. The reserve is held in a sub-account under the carrier and released when the debtor pays the invoice in full.

### `frontend/server/routers/settlementBatching.ts`

Advances participate in the weekly settlement batch. When an invoice is `funded`, the batch job marks the underlying load's settlement row as `advanced_haulpay` so it is excluded from the next driver settlement. When HaulPay reports `collected`, the residual reserve is released into the batch.

## 4. iOS Screens for the FACTORING Role

The FACTORING persona in-app is **web-first**; the iOS app delivers a **read-only dashboard with a narrow write surface** (approve/deny, mark under review). Full case management — chargeback workups, reserve adjustments, carrier onboarding interventions — stays on the web console.

Screens:

1. **Factoring Desk Dashboard.** Today's queue count, funded-today total, aging 30/60/90 summary, chargeback rate (target under 0.5%). Header: EusoHeader with the factoring title ring and the desk-operator initials. Hero tile: an ActiveCard that shows the oldest unresolved invoice needing action.
2. **Invoice Approval Queue.** A list of invoices in `submitted` or `under_review`. Each row shows carrier name, debtor name, amount, aging, credit-score badge, and a swipe-action pair: Approve (EusoBadge.success preview) and Deny (EusoBadge.warning preview). Tapping opens a detail sheet with the BOL/POD thumbnails.
3. **Chargeback Review.** Case file view: original invoice, debtor communication trail, the carrier's response, recommended disposition. The operator taps "Write off" (releases reserve), "Rebill carrier" (creates a carrier-chargeback payment), or "Hold for legal."
4. **Carrier Enrollment Status Tracker.** Pulled from HaulPay's `carrier_relationships_list`. Each carrier shows KYC stage (documents pending, Plaid linked, OFAC cleared) and has an EusoBadge indicating the readiness state.
5. **Daily Funding Report.** Funded today, by lane, by debtor, by carrier, with a floor-of-funds ticker at the top.
6. **Compliance Audit Trail.** All role actions, pulled from the `auditedFactoringProcedure` log. Filterable by operator, date, action, and entity.

## 5. iOS Screens for the Driver Interacting with Factoring

The DRIVER is the common counterparty. Driver-facing surfaces are where we compete on UX, because drivers already have five factoring apps on their phone and will uninstall whichever one is slowest.

1. **Wallet §6 Factoring Offer Card.** Shipped in iOS build 54, polished in 55. The card uses `LinearGradient.diagonal`, the hot gradient palette for "same-day advance available," and a single primary CTA "Get paid today." Amount, fee, arrival ETA are displayed in a three-row ledger below the CTA.
2. **Instant Payout Eligibility Check.** Before the CTA is enabled, the app calls `factoring.getOffer(loadId:)`; an ineligible load shows a neutral state card with the blocking reason (POD missing, debtor not on file, credit exhausted).
3. **Advance History.** A compact list that groups advances by month. Each row has a status badge (funded, collected, chargedback) and the invoice number tappable into the full detail.
4. **Quickpay TOS Acceptance.** First-time-only modal. Plain-language summary of the factor-of-record relationship, fee, and reserve. Single "I Agree" button and a "Read full terms" link.

## 6. KYC + Compliance for Factoring

- **KYC under BSA/AML.** The carrier entity (EIN, legal name, principal), beneficial ownership, and a US address. HaulPay performs the KYC; EusoTrip stores only the resulting status and document IDs (not the documents themselves).
- **OFAC screening** of carrier and broker/debtor on enrollment and on every twelve-month anniversary. A match blocks advances until manually cleared by a tier-2 operator.
- **UCC filings** on carrier receivables. When a carrier accepts HaulPay factoring, HaulPay files a UCC-1 financing statement against the carrier's accounts receivable. This is a legal prerequisite for first-position factor status.
- **Notice of Assignment rules** (FMCSA 49 CFR Part 377 territory, broadly the NOA regime). Every invoice sent to a debtor must carry the NOA, directing payment to HaulPay's lockbox. The backend generates the NOA PDF as part of the invoice package.
- **HaulPay is first-position factor; co-factoring disallowed.** Per HaulPay's TOS, a carrier cannot simultaneously factor with HaulPay and another factor. Enrollment explicitly terminates any prior factoring relationship. Our iOS onboarding asks this question and blocks until acknowledged.

## 7. Fee + Economics

- **Standard rate:** 2.5% (250 bps), configurable per carrier via the `FACTORING_STANDARD` config row and the user-override table.
- **Instant advance fee:** +0.5% (50 bps) for same-day ACH quickpay; stored under `FACTORING_QUICKPAY` with default `baseRate = 0.03` (which is the combined 3.0%, not the incremental 50 bps).
- **Chargeback reserve:** 3–10% held back at advance time, released when the debtor pays in full. The reserve is per-invoice and per-carrier; the carrier's aggregate reserve balance is surfaced in `factoring.getOverview` under `reserveBalance`.
- **Volume discounts** at $100K/mo (tier-1, -10 bps), $500K/mo (tier-2, -25 bps), $2M/mo (tier-3, -50 bps). Discounts are applied via `userFeeOverrides` with `overrideType = "rate_adjustment"`.

## 8. Risk Management

- **Debtor concentration limit.** No single broker or shipper may exceed 35% of a carrier's factored volume over a trailing 90-day window. Exceeding this triggers a tier-2 manual review on the next submitted invoice.
- **Credit bureaus.** EFS, DAT Credit, FreightWaves SONAR broker scorecard, and the internal `computeCreditScore` output are displayed side-by-side on debtor detail sheets.
- **Fraud signals.** Duplicate invoice number against the same debtor in a 30-day window; mismatched BOL-vs-POD (commodity, weight, or addresses diverge); off-manifest cargo (POD references goods not on the BOL); POD image EXIF timestamp earlier than the BOL. Any single hit flags the invoice for `under_review`; two hits auto-deny.
- **Hold codes.** `pending_verification`, `disputed`, `chargeback`, `written_off`. Each maps to a status badge and a reserved reason surface in the desk UI.

---

## 9. Settlement Flow

- **Weekly ACH.** Default settlement path for both carriers and drivers. Cutoff Friday 11:00 AM Central; funds arrive Monday.
- **Instant (1.5% fee) via Stripe Connect.** Elected in the Wallet Preferences screen. Fee is waived if advance quickpay was already taken on the same load.
- **Debit card instant via Visa Direct or Zelle** where the recipient bank supports it. FDX limits apply (typically $10K per transaction, $25K per day).

## 10. Integration Landmines

1. **`HaulPay.health` endpoint requires FACTORING role.** Per SKILL.md §13: our backend router double-calls `requireRole` somewhere on the health path, which currently blocks FACTORING-role sessions even though they should be the only ones able to hit it. This is a backend bug to **FIX**, not to front-run on iOS. The iOS client should *not* add a defensive bypass; we own the fix on the server and must ship it before build 56.
2. **Webhook signature verification.** HaulPay signs callbacks with HMAC-SHA256 over `timestamp + body` using a shared secret stored in `HAULPAY_WEBHOOK_SECRET`. Verify on every callback. Store the timestamp in a replay-prevention table with a 5-minute window; reject any repeat.
3. **Idempotency.** All HaulPay mutations accept an `Idempotency-Key` header. Generate this as a ULID from our `OfflineQueue` so that an offline-then-online double-submit from a driver's phone never double-advances. Store the response keyed by ULID for the four-hour HaulPay retention window.
4. **Docs structure mismatch.** The docs.haulpay.io site is structured as Broker HaulPay API / Carrier HaulPay API / Webhooks / Load Board API / Versioning / Client Workflows — not the `reference/quickpay` or `reference/invoices` paths we expected. Our doctrine must reference the broker-API and carrier-API pages, not the non-existent `reference/` tree.
5. **Planned-but-not-live endpoints.** HaulPay's "planned expansions" list includes: external creation of debtors and carriers, contract-type specification on invoice create, and credit-limit exposure to trusted parties. Do not code against these yet; poll the docs each release cycle to catch when they go live.

## 11. UX for the FACTORING Role (Design Integrity Non-Negotiable)

The FACTORING desk inherits the full EusoTrip primitive set. **No compromise, no stock tiles, no hardcoded SF Symbol gray fallbacks.**

- `LinearGradient.diagonal` on every hero.
- `palette` only — no hex literals.
- `EusoHeader` on every screen, with the desk-operator tint ring.
- `EusoBadge` with the following semantic mapping:
  - `.warning` — pending verification.
  - `.info` — under review.
  - `.hot` — same-day advance available (the only hot-palette surface in the factoring role; use sparingly to preserve signal).
  - `.success` — funded.
  - `.neutral` — historical records and write-offs.
- `ActiveCard` for advance-offer heroes on both the factoring-desk dashboard (oldest unresolved) and the driver wallet (current offer).
- **No stock SF Symbol gray tiles.** Brand tints rotate per carrier, per debtor, per invoice type — the factoring desk should feel like a trading terminal, not a CRM table.
- **Night and Afternoon registers** both supported. The night register gets the dark-quicksilver palette; afternoon uses the noon-sun accents. Switching is driven by system appearance plus a local-time override.

## 12. iOS-Specific APIs for Factoring

Wired on `EusoTripAPI.shared.factoring`:

- `getOffer(loadId:)` — **live**, wired to the backend that wraps HaulPay.
- `accept(loadId:offerId:)` — **live**, triggers the HaulPay advance request with an Idempotency-Key ULID from OfflineQueue.
- `listPending()` — **add if missing**; returns the factoring-desk queue of `submitted` and `under_review` invoices.
- `approve(invoiceId:)` — **add if missing**; transitions to `approved` and schedules funding. Enforces the tier-1/tier-2 amount split server-side.
- `deny(invoiceId:reason:)` — **add if missing**; transitions to `disputed` with a structured reason code (one of: `missing_document`, `mismatched_bol_pod`, `debtor_credit_insufficient`, `duplicate_invoice`, `off_manifest`, `other`). Reason is mandatory for the audit trail.

## 14. Reporting

- **Daily advance volume** by carrier, lane, and debtor.
- **Chargeback rate** with a 0.5% target and an in-product alert when the trailing-30-day rate exceeds 0.8%.
- **Aging buckets** 30 / 60 / 90 / 90+, with per-debtor drilldown.
- **Reserve balance** per carrier, reconciled daily against HaulPay's reserve ledger.
- **Funding velocity** — median hours from `submitted` to `funded`, split by quickpay vs. standard.
- **Denial rate** and **denial reason distribution**, to surface recurring document-capture issues that a driver UX tweak could fix.

## 15. Future

- **Multi-factor marketplace.** EusoTrip evolves from a single HaulPay integration into a marketplace where HaulPay, a second factor, and a third compete per load. Carriers pick a default factor in onboarding and get an offer-comparison view on each load. This requires abstracting the `factoring.getOffer` response into a list and adding `offerId` to the accept call — already in place.
- **Auto-factor on load acceptance (opt-in).** Carrier toggles a Wallet setting and every load they accept automatically posts for factoring upon POD. The "Get paid today" card becomes a "Funding in progress" card from delivery forward.
- **Dynamic rate per lane and broker.** The fee is priced per the combination of lane risk, broker payment history, and carrier history — all computed by the internal credit-score model. Rates move in real time but are disclosed before acceptance.
- **Co-factoring with regulatory approval.** If HaulPay's TOS and the broader factoring industry evolve to permit split positions, EusoTrip is positioned to broker the split.

## Closing

The FACTORING role is a money-movement persona, firewalled from operational routers, represented in `ROLES.FACTORING`, and served by a dedicated pre-built `factoringProcedure` / `auditedFactoringProcedure` / `isolatedFactoringProcedure` set in `frontend/server/_core/trpc.ts`. It wraps HaulPay's Bearer-token REST API at `https://api.haulpay.io/v1/external_api/`, surfacing invoice submission, quickpay advances, webhook-driven status sync, and a platform-internal credit-score layer that adds value on top of HaulPay's own debtor credit.

The role ships to iOS as a read-only-plus-approve dashboard that must match the EusoTrip primitive palette without compromise. The driver-side counterpart — the Wallet Factoring Offer card — is the hero surface where we win or lose the factoring competition, and it is already live in iOS builds 54 and 55.

This doctrine is canonical. Treat any deviation as a regression.
